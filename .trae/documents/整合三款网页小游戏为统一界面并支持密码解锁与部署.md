## 实施目标
整合三款网页小游戏到统一主页，提供密码解锁（仅限“2.四色困境”“3.水坝决堤”），保持每个游戏独立维护与可单独修改，兼容移动与桌面，并可静态部署以支持课堂并发使用。

## 变更范围
仅新增主页 `index.html` 与少量主页资源，不改动三个现有游戏文件。主页通过 `iframe` 加载游戏确保彼此隔离。

## 具体实施步骤（对应 L75–87）
1. 创建 `index.html`（统一界面）：
   - 顶部标题与说明区域。
   - 三张游戏卡片，分别代表三款游戏；展示封面、简介、进入按钮。
   - 使用 Tailwind CDN，与现有风格一致。
2. 为“四色困境”“水坝决堤”添加锁定与密码输入：
   - 卡片默认显示锁定覆盖层与密码输入框、解锁按钮。
   - 前端以 SHA-256 对输入密码计算摘要，与预置摘要常量比对；成功后移除锁定并允许进入。
   - 在 `sessionStorage` 记录 `game2Unlocked`、`game3Unlocked`，同一会话刷新持续解锁。
3. 设计 `iframe` 区域（同页展示游戏）：
   - 点击“进入”后在主区域以 `iframe` 加载对应游戏（`1.九星迷阵.html`、`2.四色困境.html`、`3.水坝决堤.html`）。
   - 提供控制：返回主页、全屏、在新窗口打开。
   - 显示“加载中”指示器，游戏加载完成后隐藏。
4. 响应式样式：
   - 移动端单列卡片；平板/桌面两到三列自适应。
   - `iframe` 容器宽度自适应、固定高宽比（如 4:3/1:1），移动端提供“全屏”以提升体验。
5. 状态保存：
   - 使用 `sessionStorage` 保存解锁状态；退出浏览器会话后重置。
   - 可选：`localStorage` 切换为持久保存（默认先用 `sessionStorage`）。
6. 交互与无障碍：
   - 表单输入与按钮支持键盘操作；错误提示清晰。
   - 加载指示与失败重试；触屏与点击均可操作。
7. 测试：
   - 设备：移动端（iOS/Android 浏览器），桌面端（Chrome/Edge）。
   - 用例：密码正确/错误、刷新后保持会话解锁、多用户并发访问、全屏与新窗口打开、返回主页切换。
   - 验证三款游戏在 `iframe` 下 `window.onload` 正常执行、互不干扰。

## 安全与可维护性
- 密码仅存摘要常量，不保存明文；前端校验适用于课堂场景。若需防直链，可后续在锁定游戏页加入令牌校验（独立改动）。
- 游戏文件保持原名与路径；若未来重命名，更新主页 `iframe src` 即可。

## 交付与部署
- 交付：`index.html` 与必要静态资源。
- 部署：静态托管（如 GitHub Pages），无需后端，天然支持并发。
